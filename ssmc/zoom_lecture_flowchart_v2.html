STEP 01 강의 개설 요청 변경 없음 강사 / 운영자
📝 플랫폼 UI에서 강의명, 시작일시, 수강생 목록 입력 후 개설 요청
💡 이 시점에 Zoom 계정은 미배정 상태. 다음 단계에서 DB 조회 후 배정.
STEP 02 Zoom 계정 조회 / 배정 ▲ 구조 변경 Server (Backend)
① Primary — DB 조회 (필수)
🗄️ 해당 과정에 배정된 zoom 계정 조회
zoom_accounts.assigned_course_id = {courseId}
🔒 중복 배정 방지: assigned_until 체크 → 과정 기간 내 다른 과정에 재배정 불가
🔄 미배정 계정(assigned_course_id IS NULL) 중 풀(pool) 대기 계정 자동 선택
② Secondary — Zoom API 보조 체크
GET /v2/users/{userId}/meetings?type=live
// 현재 live 미팅 유무로 사용 중 여부 보조 확인
// Zoom은 미팅 생성 수 제한 없음 → upcoming으로 판단 불가
// DB 배정이 Primary — API는 double-check 용도
⚠️
Zoom API 단독으로 배정 판단 금지. Zoom은 미팅 생성 수 제한 없음 → upcoming 수로 사용 여부 판단 불가. live 체크는 보조 참고용.
과정-계정 매핑은 반드시 DB가 Source of Truth
STEP 03 Access Token 발급 변경 없음 Server → Zoom OAuth
POST zoom.us/oauth/token
grant_type=account_credentials
account_id={단일 Account ID} // .env — 앱 1벌
🔑 발급된 access_token은 1시간 유효. 강의 시작 시 재발급 로직 필요.
STEP 04 미팅 생성 변경 없음 Server → Zoom API
POST /v2/users/{zoom계정 email}/meetings
Authorization: Bearer {access_token}

type: 2 // 예약 미팅
topic: "{강의명}"
start_time: "{시작일시 ISO8601}"
duration: {분}
STEP 05 응답값 DB 저장 ▲ 필드 추가 Server → DB
Zoom API 응답 → DB 저장 필드
// ── 기존 필드 ──────────────────
meeting_id: "85746523198"
join_url: "https://zoom.us/j/..."
password: "abc123"

// ── 신규 추가 필드 ──────────────
start_url: "https://zoom.us/s/...?zak=..." // ★ 호스트 입장 필수
course_id: "{과정ID}" // 레코딩 귀속 추적
lecture_id: "{강의ID}" // 레코딩 귀속 추적
✅
start_url은 ZAK 포함된 호스트 전용 URL. 미팅 시작 시 만료되므로
강의 시작 버튼 클릭 시점에 아래 API로 재조회 권장 (방안 A)
방안 A — 강의 시작 시점 start_url 재발급
GET /v2/meetings/{meetingId}
// 강사가 "강의 시작" 버튼 클릭 시 호출
// 응답의 start_url을 딥링크 파라미터로 전달
// → zoomus:// 실행
⚡ DB에 저장된 start_url은 백업/디버그용으로만 사용.
실제 입장에는 항상 재조회값 사용 권장.
STEP 06 녹화 완료 Webhook 처리 ▲ 귀속 추적 추가 Zoom → Server Webhook
Webhook Event: recording.completed
🔐 Secret Token 검증 → 유효하지 않으면 즉시 reject
🔍 payload의 meeting_id → DB에서 course_id, lecture_id 매핑 조회
💾 share_url, recording_files → DB 저장 (S3 이관 없음 — 확정)
🎯
course_id / lecture_id 저장(STEP 05)이 없으면
어느 강의 녹화인지 추적 불가. 두 필드는 STEP 05에서 반드시 저장.

S2

동적 매핑 (풀 운용)

TRIGGER

현재 live 미팅 유무 확인 (보조)

ZOOM OAUTH
배정된 계정의 Token 조회/갱신
만료 시 Refresh Token으로 재발급
ZOOM API
미팅 생성
POST /v2/users/{동적배정계정}/meetings
DB SAVE + 과정 종료 후 반납 스케줄
meeting_id, join_url, start_url 저장
assigned_until 자동 체크 → 풀 반납 처리
✅ 장점
·계정 수 절감 (비용 효율)
·과정 증가 시 계정 추가 불필요
·유휴 비용 최소화
❌ 단점
·동시 몰림 시 배정 실패 가능
·레코딩 귀속 추적 복잡
·반납 타이밍 오류 → 중복 배정 위험
·구현 복잡도 높음
·즉흥형 과정 "바로 시작" 불가
📊 항목별 비교
비교 항목 S1 1:1 고정 매핑 S2 동적 매핑
계정 수 과정 수 = 계정 수 (1:1) 동시 강의 수 기준으로 절감 가능
라이선스 비용 보통 유휴 기간도 비용 발생 절감 동시 사용 수만큼만
구현 복잡도 낮음 단일 경로, 분기 없음 높음 배정/반납/Lock/스케줄러
운영 안정성 높음 배정 실패 없음 리스크 동시 몰림 시 배정 불가
레코딩 귀속 명확 계정 = 과정 1:1 복잡 course_id/lecture_id 추적 필수
즉흥형 과정 즉시 가능 불가 가용 계정 없으면 보류
계정 추가 시 OAuth 수동 동의 필요 (1회) OAuth 수동 동의 + 풀 로직 수정
장애 영향 범위 해당 과정 1개만 영향 풀 전체 로직 오류 시 전체 영향
스케줄러 필요 불필요 필요 (계정 반납 자동화)
추가 구현 항목 없음 Lock 처리, 반납 스케줄러, 알림, 충돌 감지

🎯 권장 방향 — 시나리오 1 + 예비 계정

시나리오 1이 권장입니다. 단, 유연성을 위해 예비 계정 1~2개를 풀로 운용하는 변형을 제안합니다.

핵심 근거: 시나리오 2의 "배정 실패" 상황은 기술 문제가 아니라 운영 사고입니다. 강의 시작 직전 배정 불가 알림이 뜨는 상황에서 대응 프로세스를 만들어야 하고, 그 비용이 라이선스 절감액보다 높습니다.

예비 계정 구조:
· 과정 N개 → 유료 계정 N개 (고정 매핑)
· 예비 계정 1~2개 → 긴급 과정 추가, 계정 장애 시 즉시 교체
· 동적 배정 로직 없이 운영자가 수동으로 매핑만 변경

Zoom Pro 계정 기준 월 ~$15/계정. 예비 1개 추가 비용 $15로 동적 배정 로직 전체 구현·유지보수 비용을 대체합니다.

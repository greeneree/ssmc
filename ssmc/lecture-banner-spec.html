⚠ 검토 의견 (기획안과 다른 관점)

DB 기준 종료 트리거는 맞아. 근데 지금 버그의 본질은 트리거 문제만이 아니야.
9시 30분 강의가 13시 19분에도 노출된 이유 — 트리거 없음 + 강사에게 자기 강의 상태 피드백이 없음 이 두 개가 같이 터진 거야.

운영에서 풀어야 할 문제라는 건 동의해. 근데 강사가 자기 강의가 "아직 열린 상태"인지 인지할 수 있으면 운영 개입 빈도 자체가 줄어. 역할별 상태 문구는 UX가 아니라 운영 비용 절감 장치야.

기획안 DB 종료시간 기준으로 배너 비노출. 초과케이스는 운영에서 처리.
추가 의견 DB 기준 트리거 유지 + 역할별 상태 문구로 강사 자기인지 루프 추가 → 운영 에스컬레이션 감소.
공통 즉시강의 웹훅 로직 건드리지 않음. 예약강의만 적용.
TRIGGER LOGIC

  • 강사 설정 시간(30분/1시간) 초과 시 자동 종료
  • 기존 로직 그대로 유지 — 변경 없음

개발자 확인 요청 사항
1
DB 종료 예정 시각 저장 형태 — 원래 종료 시각 저장인지, 버퍼(+30분) 포함 시각이 이미 저장돼 있는지. 버퍼 계산을 서버에서 할지 DB에서 할지 결정 필요.
2
강사 입장 여부 상태값 — 현재 웹훅 외 별도 상태값 존재하는지. 없으면 Zoom 웹훅에서 강사 join 이벤트 수신 시 상태값 저장하는 로직 추가 필요.
3
배너 1개 노출 기준 — 현재 시각 기준 "진행 중인 강의" 우선인지, "다음 예정 강의" 우선인지 정의 필요. 진행 중 강의가 있으면 해당 강의 우선 표시 권장.
4
즉시강의 로직 영향 없음 확인 — 이번 변경은 예약강의에만 적용. 즉시강의 웹훅 기반 종료 로직 변경 없음.
